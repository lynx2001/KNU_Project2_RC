SPICE(Automotive SPICE)는 자동차 소프트웨어 개발 프로세스의 성숙도를 평가하는 국제 표준입니다. A-SPICE 심사에서 주로 기준이 되는 핵심 영역(VDA Scope, 16개 프로세스)은 V-모델 개발 주기를 따르며, 단계별로 요구되는 산출물(Work Products / Information Items)이 명확히 정의되어 있습니다.


# **[각 주요 그룹 및 단계별 핵심 산출물]**


## 시스템 엔지니어링 프로세스 (System Engineering, SYS)


→ 차량 전체 시스템의 큰 기능(하드웨어 + 소프트웨어)을 정의하고 설계 및 검증하는 단계입니다.


**SYS.2 시스템 요구사항 분석:**


시스템 요구사항 명세서 (System Requirements Specification, SRS)


요구사항 추적성 매트릭스 (Traceability Matrix)


**SYS.3 시스템 아키텍처 설계:**


시스템 아키텍처 설계서 (System Architecture Document, SAD)


시스템 인터페이스 정의서 (Interface Control Document)


**SYS.4 시스템 통합 및 통합 검증:**


시스템 통합 테스트 명세서 (System Integration Test Specification)


시스템 통합 테스트 결과 보고서 (System Integration Test Report)


**SYS.5 시스템 적격성 검증:**


시스템 적격성 검증 명세서 (System Qualification Test Specification)


시스템 적격성 검증 결과 보고서 (System Qualification Test Report)


## 소프트웨어 엔지니어링 프로세스 (Software Engineering, SWE)


→ 시스템 아키텍처를 바탕으로 실제 소프트웨어를 설계, 구현하고 꼼꼼히 테스트하는 단계입니다.


**SWE.1 소프트웨어 요구사항 분석:**


소프트웨어 요구사항 명세서 (Software Requirements Specification, SWRS)


소프트웨어 요구사항 분석 보고서 (요구사항 타당성 및 리스크 검토 내용 포함)


**SWE.2 소프트웨어 아키텍처 설계:**


소프트웨어 아키텍처 설계서 (Software Architecture Document, SWAD)


소프트웨어 인터페이스 명세서


**SWE.3 소프트웨어 상세 설계 및 단위 구현:**


소프트웨어 상세 설계서 (Software Detailed Design, SWDD)


소스 코드 (Source Code)


**SWE.4 소프트웨어 단위 검증:**


단위 테스트 명세서 (Unit Test Specification)


단위 테스트 결과 보고서 (정적 분석 결과 및 MISRA C 등 코딩 룰 준수 보고서 포함)


**SWE.5 소프트웨어 통합 및 통합 검증:**


소프트웨어 통합 테스트 명세서 (SW Integration Test Specification)


소프트웨어 통합 테스트 결과 보고서 (SW Integration Test Report)


**SWE.6 소프트웨어 적격성 검증:**


소프트웨어 적격성 검증 명세서 (SW Qualification Test Specification) - 테스트 기법, 환경, 통과 기준 등 명시


소프트웨어 적격성 검증 결과 보고서 (SW Qualification Test Report)


## 관리 및 지원 프로세스 (Management & Support, MAN/SUP)


→ 개발 사이클 전반에 걸쳐 모니터링, 버전 관리, 품질 검토를 수행하는 단계입니다.


**MAN.3 프로젝트 관리:**


프로젝트 관리 계획서 (Project Plan) - 일정, 예산, 리스크 관리 포함


프로젝트 진행 상태 보고서 (Status Report)


**SUP.1 품질 보증:**


품질 보증 계획서 (QA Plan)


품질 보증(QA) 감사 결과 보고서 및 부적합 사항 보고서


**SUP.8 형상 관리:**


형상 관리 계획서 (Configuration Management Plan)


베이스라인(Baseline) 및 버전 관리 대장


**SUP.9/SUP.10 문제 해결 및 변경 관리:**


이슈/결함 보고서 (Issue/Defect Report)


변경 요청서 (Change Request, CR)

## 1. 요구 사항 및 설계 사양서 (The "What" & "How")


단순한 기능 나열이 아니라, '무엇을(What)' 할지 정의하고 '어떻게(How)' 구현할지 기술하는 단계입니다.

- **요구 사항 사양서(SRS):** 사용자 관점의 비즈니스 로직, 기능적/비기능적 요구 사항(성능, 보안 등)을 명확히 합니다.
- **설계 사양서(SDS):** 시스템 아키텍처, 데이터베이스 스키마, UI/UX 흐름, 외부 연동 규격 등을 포함합니다.
- **체크포인트:** "이 문서만 보고 개발자가 코딩을 시작할 수 있는가?"를 자문해 보세요.

## 2. R&R 정립 (The "Who")


누가 무엇을 책임지는지 명확하지 않으면 프로젝트 후반부에 병목 현상이 발생합니다.

- **RACI 매트릭스 활용:**
	- **Responsible (실무자):** 실제로 일을 수행하는 사람.
	- **Accountable (책임자):** 최종 의사 결정 및 결과에 책임을 지는 사람 (1명 권장).
	- **Consulted (협의자):** 업무 수행 시 조언을 주는 전문가.
	- **Informed (보고 대상):** 진행 상황을 알아야 하는 이해관계자.
- **체크포인트:** 의사 결정 라인이 명확한지, 업무 누락(Gap)이나 중복(Overlap)은 없는지 확인합니다.

## 3. 예산 산정 (The "How Much")


앞선 요구 사항과 R&R을 바탕으로 현실적인 비용을 도출합니다.

- **인건비 (M/M):** 투입 인력의 등급과 기간에 따른 산출.
- **인프라 및 솔루션:** 클라우드(AWS/Azure), SW 라이선스, 하드웨어 구매 비용.
- **예비비 (Contingency):** 예상치 못한 변수나 요구 사항 변경을 대비한 10~15% 수준의 여유 예산.
